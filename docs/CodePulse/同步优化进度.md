## 1.1 扫盘间隔与 debounce

**要解决的问题**
无 hook 时，磁盘兜底最长约 8s 才刷新；watch 后还要等 600ms debounce，体感「慢半拍」。

**改动**

- `packages/local-server/src/session-sync.ts`
  - `STEADY_INTERVAL_MS`: `8000` → **`3500`**
  - `WATCH_DEBOUNCE_MS`: `600` → **`200`**
  - `BOOT_OFFSETS_MS`: `[0,800,2000,4000,6000]` → **`[0,500,2000]`**（少排队）

**结果**
稳态最坏延迟约减半；watch 路径约快 400ms；启动少 2 次无效全量扫。

---

## 1.2 进程探测缓存

**要解决的问题**
每次 `syncCodex` 可能跑 `tasklist`/`pgrep`（超时 3s），Windows 上拖长整次扫盘。

**改动**

- `isCliProcessAlive`：按 kind 缓存 **3s**；探测超时 **1s**

**结果**
8s/3.5s 稳态环上重复扫不再反复打 tasklist；失败仍回退为「假定存活」（保持原语义）。

---

## 1.3 status IPC 合并

**要解决的问题**
工具风暴时每个 `ingest` 立即 `broadcast(status)`，渲染与主进程被打满。

**改动**

- `apps/desktop/src/main/index.ts` `wireHub`：status 推送 **32ms** 合并（取最新 snapshot 一次刷出）

**结果**
忙时 IPC 次数显著下降；单次状态延迟最多 +32ms（可接受）。Hub 本体测试行为不变。

---

## 1.4 托盘语义 diff

**要解决的问题**
`trayStatusKey` 序列化整棵 agent（含 token、lastEventAt），工具每一下都重建托盘菜单。

**改动**

- `trayStatusKey` 仅：`overall` + 每 agent `{ type, state, unread, workspace, model }`

**结果**
上下文/额度抖动不再触发菜单重建；状态/未读/项目变化仍更新托盘。

---

## 1.5 Hook soft-reset（准确）

**要解决的问题**  
`codex-usage.js` 对过期 limits **delete**；server soft-reset 为 **0%**。Hook 不发 limits 时 hub 粘住重置前高 %。

**改动**

- 过期 limits → **used_percent: 0**，保留 resets_at（与 `quota-watcher` 一致）
- 更新相关测试期望

**结果**  
重置后 hook 路径也会把额度打到 0%/可刷新，而不是静默消失导致粘旧值。

---

## 1.6 过期窗口禁止旧高 %（准确）

**要解决的问题**  
`mergeRateLimitWindow` 在 `windowExpired` 时 last-write，soft-reset 0% 可被另一份过期 88% 盖回。

**改动**

- 同 past `resetsAt` 且已过期：**usedPercent 取 min**（0 粘住）

**结果**  
过期后以 soft-reset 为准，直到出现**新的** future `resetsAt` 窗口。

## 2.1 SQLite 跳过 internal 扫盘事件

**要解决的问题**  
每次 session-sync / quota-refresh 的 token_snapshot 都同步写 SQLite，工具风暴时放大主线程压力。

**改动**

- main persist 跳过 `internal.quotaRefresh` / `internal.sessionSync`（lifecycle 仍写）

**结果**  
扫盘/额度轮询不再灌库；用户可见事件仍落盘。

---

## 2.2 统一 Codex reader

**要解决的问题**  
session-sync fallback 与 quota-watcher 读法不一致：fallback 可能带 raw 过期 limits / total_token_usage，绕过 soft-reset。

**改动**

- 主路径：`readCodexQuotaTokenFromFile`（4MB 尾 + soft-reset）
- `readCodexTokenFallback`：仅 `last_token_usage` 上下文，**不**附 rateLimits
- `CODEX_TAIL = 4MB` 与 watcher 对齐

**结果**  
额度只走 soft-reset 路径；fallback 只补上下文。

---

## 2.3 session-sync 按 bucket 共享额度

**要解决的问题**  
账号级共享额度时清空 `quotaBuckets`，双 meter（main + Spark）丢失。

**改动**

- `pickSharedCodexAccountQuotaFamilies`：分别选 main / spark 最佳快照
- `withSharedCodexAccountQuota`：叠加双方 rateLimits 到 `quotaBuckets`，top-level 按项目是否 Spark 选主条
- 比较器优先 **周额度**（七天 resetsAt / used%），5h 仅作弱平手

**结果**  
双账号 meter 保留；项目卡片显示与账号类型匹配的主额度。

---

## 2.4 Claude 模型感知周额度 + accuracy

**要解决的问题**

1. OAuth 响应含 `seven_day_opus` / `seven_day_sonnet` 时固定 first-hit，与当前模型不符
2. statusline 无官方 `used_percentage` 时仍标 `exact`，estimated 数学被当精确

**改动**

- `normalizeClaudeRateLimitsPayload(raw, preferredModel?)`：opus/sonnet 优先，否则 generic / 最高 used%
- `ClaudeQuotaSnapshot.rawFamilies` + 缓存多家族窗口；`mergeClaudeContextWithQuota(..., model)` 重选
- adapter：`exact` 仅当官方 used_percentage + window size 同时存在
- state-machine：`keepExactContext` 时 usage 字段也不被 estimated 覆盖

**结果**  
周额度跟模型族；上下文 accuracy 更诚实，exact 不被估算覆盖。

---

## 2.5 统一 now 时钟

**要解决的问题**  
多个 meter/tile 各自 `useNow(500)` → N 个 setInterval，忙时渲染抖动。

**改动**

- `useNow`：按 interval 共享单一 clock + 订阅表
- `UI_REFRESH_INTERVAL_MS`：`500` → **`1000`**
- 隐藏标签页不 tick

**结果**  
N 订阅者共用 1 个 timer；刷新更稳、更省。

## 3.1 按源脏标记扫盘

**要解决的问题**

任意 Codex/Grok/Claude 文件变化都会触发三种来源的全量扫描；单一 CLI 高频写盘时会连带读取另外两套目录与额度。

**改动**

- `SessionSyncService` 的 watcher 与来源一一绑定
- debounce 窗口内累计 `dirtySources`，到点只调度变化过的来源
- `syncNow(sources?)` 支持定向扫描；boot / steady / focus 仍保留全来源兜底

**结果**

watch 热路径不再连带扫描无关 CLI；多个来源在同一 200ms 窗口变化时仍合并为一轮同步。

---

## 3.2 focus 防抖 sync

**要解决的问题**

托盘打开、Electron `focus`、窗口创建安全补扫和渲染层 bootstrap 可能同时触发多轮全量扫描。

**改动**

- 新增 `FocusSyncScheduler`，以 **180ms** 合并聚焦请求
- 同一批调用者共享一次异步扫描结果；扫描期间的新请求合并为一次 trailing scan
- 窗口 `focus`、托盘打开、IPC 主动同步统一走同一调度器

**结果**

回到窗口仍会及时补齐磁盘状态，同时消除一次聚焦动作引发的重复扫盘堆叠。

---

## 3.3 路径 realpath

**要解决的问题**

Hook 与磁盘元数据可能分别上报符号链接/目录联接路径和真实路径，导致同一项目出现两张卡片。

**改动**

- 新增异步 `WorkspacePathResolver`
- Hook API 与 Codex/Grok/Claude 扫盘入口都在进入 Hub 前解析 `realpath`
- 成功结果缓存 **30s**；失败结果缓存 **1s** 后重试；不存在或不可访问的路径保留原值

**结果**

可解析的 symlink/junction 别名统一为真实项目根路径；工具事件风暴不会为同一路径反复访问文件系统。

---

## 3.4 多项目虚拟列表

**要解决的问题**

每个 agent 面板一次渲染全部项目卡片，多项目时 DOM、布局和状态刷新成本线性增长。

**改动**

- 项目数 **>8** 时切换为窗口化列表
- 仅挂载可视区 + **280px overscan** 的项目卡片
- `ResizeObserver` 记录动态行高，真实高度替换初始估算；短列表保持原渲染路径

**结果**

长列表滚动时 DOM 数量受视口约束；不同语言、未读按钮等高度变化不会造成卡片重叠。

## 4.1 Linux AppImage 发布链路

**要解决的问题**

`electron-builder` 虽声明了 AppImage target，但仓库没有独立的 Linux 构建命令，发版流程也只产出 Windows 与 macOS 安装包。

**改动**

- 根目录与桌面包新增 `dist:linux`，固定构建 Linux x64 AppImage
- AppImage 产物统一命名为 `CodePulse_<version>_linux-x86_64.AppImage`
- Release workflow 新增 Ubuntu 构建任务，校验产物存在、可执行且能响应 `--appimage-version`
- 发布任务汇总 `.AppImage` 到与 Windows、macOS 相同的 GitHub Release
- Linux/macOS 检查更新时只打开 Release 页面，不会误启动 Windows NSIS 安装包

**结果**

本地可执行 `pnpm dist:linux` 生成 AppImage；tag / 手动发版会与 Windows、macOS 并行生成并发布 Linux 产物。

---

## 4.2 发布验证与原生 ABI 稳定性

**要解决的问题**

CP1 请求和配网轮询依赖的 timer 被 `unref`，Node 测试进程可能在 Promise 完成前退出；同时安装阶段把 `better-sqlite3` 提前切到 Electron ABI，导致 Node 存储测试静默跳过。

**改动**

- 仅后台扫描 timer 保留 `unref`；请求超时与配网轮询 timer 保持事件循环存活
- `pnpm test` 显式恢复 Node ABI，`dev/start` 显式切换 Electron ABI，打包仍由 builder 重建
- SQLite ABI 不匹配不再 `skip`，直接作为测试失败暴露
- WebSocket 测试使用显式 `ws` 客户端，兼容项目声明支持的 Node 20
- Release workflow 新增 Node 20/22 独立验证矩阵，平台打包任务统一依赖验证结果

**结果**

Node 20 与 Node 22 均实现 **259 passed / 0 failed / 0 cancelled / 0 skipped**；Electron ABI、Linux AppImage 构建以及构建后切回 Node ABI 均验证通过。
